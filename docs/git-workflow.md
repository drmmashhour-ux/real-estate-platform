# Safe Git workflow (BNHub / LECIPM)

This repo uses **Husky**, a **pre-commit hook**, **lint-staged**, and **root `.gitignore`** so teams avoid broken pushes and bloated history.

---

## What not to commit

| Never commit | Why |
|--------------|-----|
| `node_modules/` | Restored with `pnpm install`; huge and platform-specific |
| `.next/`, `dist/`, `out/` | Build outputs; regenerated by `pnpm build` |
| `coverage/`, `.tmp/`, `.cache/` | Generated or local scratch |
| `.env`, `.env.*` | Secrets (exception: **`.env.example`** is tracked on purpose) |
| `*.log` | Noise and possible leaks |
| `.DS_Store` | macOS metadata |
| Any file **> 50MB** | GitHub rejects large blobs; use storage/CDN instead |

---

## Branching

1. From latest `main` (or your team’s default branch):  
   `git fetch origin && git checkout main && git pull`
2. Create a feature branch:  
   `git checkout -b feature/short-description`
3. Open a PR when ready; avoid committing directly to `main` if your team uses review.

---

## Committing safely

1. **Install hooks** (once per clone): `pnpm install` runs `prepare` → Husky installs `.husky/` hooks.
2. Stage only source: `git add path/to/files` (avoid `git add .` until you’ve run a build and confirmed ignores).
3. **Commit**: `git commit -m "type: message"`  
   - Pre-commit checks forbidden paths and **50MB** limit.  
   - **lint-staged** runs `scripts/git/guard-staged.mjs` on staged files (same rules, per file).
4. **Push**: `git push -u origin your-branch`

Helper scripts (root `package.json`):

- `pnpm run git:check` — status + unstaged names  
- `pnpm run git:size-check` — repository object size summary  
- `pnpm run git:clean` — re-sync index from disk respecting `.gitignore` (use when something was wrongly tracked; review before committing)

---

## Fixing “large file” or push errors

1. **Do not commit** the large file; use Git LFS only if your org standardizes on it (this doc assumes **no** LFS).
2. If it was **already committed**: remove from history with `git filter-repo` (or team-approved tooling), then force-push coordinated with the team. See also `docs/git-rules.md` if present.
3. If it was **only staged**: `git reset HEAD -- path/to/file` and add the path to `.gitignore` or move the asset to object storage.

## Reinstalling dependencies

From the monorepo root:

```bash
pnpm install
```

If lockfile or modules look corrupt:

```bash
rm -rf node_modules apps/*/node_modules packages/*/node_modules services/*/node_modules modules/*/node_modules
pnpm install
```

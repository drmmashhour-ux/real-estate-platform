# Feedback loop

Structured learning from every conversation. **Speed of learning** beats volume of features.

---

## After each demo (or serious sales call)

Ask **3–5** of these (not all as a rigid survey—natural conversation):

1. **What was unclear?** (copy, flow, terminology)  
2. **What would you actually use** in your week?  
3. **What’s missing** for you to try it for real?  
4. **What felt confusing** or “too heavy”?  
5. **Would you use this?** If not, what would need to change?

Add one open-ended: **“What’s the first thing you’d want to do on day one?”**

---

## Store feedback simply

Use whatever you’ll actually maintain:

- **Shared doc** (Google Doc / Notion) with columns: date, name, segment, notes, follow-up.  
- Or **spreadsheet**: one row per call, “themes” column for tags.

**Minimum:** date + who + 5 bullet takeaways + **one** prioritized follow-up.

---

## Categorize every item

| Category | Examples | Typical action |
|----------|----------|----------------|
| **UX issues** | Can’t find button, too many steps | Copy, layout, defaults |
| **Missing features** | Integration, report, role | Roadmap; batch similar asks |
| **Bugs** | Error, wrong data | Triage; hotfix if blocking trust |
| **Confusion** | Wrong mental model | Rename, onboarding tooltip, demo script tweak |

Tag rows so weekly review is fast.

---

## Weekly feedback review

1. **Cluster** similar notes (same confusion = one fix).  
2. Pick **top 3** themes for [ITERATION_PLAN](./ITERATION_PLAN.md).  
3. Close the loop: tell users who gave input when you ship something they asked for (builds trust).

---

## Rules

- **No feedback hoarding**—if it’s not written down, it didn’t happen.  
- **Distinguish** one loud opinion vs **three people** saying the same thing—prioritize the latter.  
- **Bugs** before speculative features when trust is on the line.

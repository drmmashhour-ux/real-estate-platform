# Git rules for the BNHub / LECIPM monorepo

For the full safe workflow (hooks, helper scripts, troubleshooting), see **`docs/git-workflow.md`**.

This repository is configured so **dependencies and build artifacts are never committed**, and **large files are blocked before they reach GitHub** (100 MiB hard limit on GitHub; we enforce **50 MiB** on commit).

## What not to commit

- **`node_modules/`** — install with npm/pnpm at the repo root (workspaces).
- **`.next/`** — Next.js build output; regenerated by `npm run build` in each app.
- **`out/`** — Next static export output (when used).
- **`dist/`** — TypeScript or bundler output under services/packages; build in CI or before publish.
- **`build/`**, **`.turbo/`**, **`.cache/`**, **`.tmp/`** — caches and intermediates.
- **`.env`, `.env.local`, secrets** — use `.env.example` / `.env.production.example` as templates only (those are whitelisted in `.gitignore`).
- **Large binaries** — prefer CDN, package registry, or Git LFS only when the team explicitly chooses LFS.

## Installing dependencies

From the repository root:

```bash
pnpm install
```

Run app or service scripts via workspaces, for example:

```bash
pnpm dev
pnpm build:web
```

## Hooks (Husky + lint-staged)

On every commit:

- Staged paths are checked for forbidden folders (`node_modules`, `.next`, `dist`, etc.).
- Staged files larger than **50 MiB** are rejected.

If a commit is rejected, fix the issue, then commit again.

## Fixing “large file” or accidental `node_modules` commits

If something was already committed:

1. Remove it from the index and ensure `.gitignore` covers it:

   ```bash
   git rm -r --cached path/to/bad-folder
   ```

2. If the large blob exists **anywhere in branch history** you intend to push, rewrite history (e.g. `git filter-repo`) or reset/rebase the offending commits. GitHub rejects pushes if **any** new commit on the wire contains a blob over 100 MiB.

3. For a full index refresh (nuclear option, use with care):

   ```bash
   git rm -r --cached -f .
   git add .
   ```

4. After history rewrites, push with:

   ```bash
   git push --force-with-lease
   ```

## Correct workflow

1. Pull latest, `npm install` when lockfiles change.
2. Develop locally; keep `node_modules` and `.next` untracked.
3. Commit only source, config, and intentional assets.
4. Let CI (or local `npm run build`) produce `dist` / production bundles.
